# Google Photos Backup & Sync Bot

Локальный Python-скрипт для автоматизации резервного копирования фотографий и видео из Google Photos с последующей умной очисткой облака.

Скрипт реализует стратегию **"Горячего и холодного хранения"**: все новые и старые фото скачиваются на локальный диск, но из облака удаляются только файлы старше определенного порога (например, 180 дней). Это позволяет сохранить недавние фото в галерее на телефоне для быстрого доступа, освобождая при этом место в Google Диск.

## Отчетность по манифесту

- **Автоматизированный экспорт**: Скачивает блоки фото и видео через браузерную автоматизацию (Playwright), эмулируя действия пользователя.
- **Умная сортировка**: Раскладывает скачанные файлы по папкам по годам (например, `Фото Google 2024`), определяя дату из имени файла или EXIF-метаданных.
- **Надежность и защита от потери данных**: Ведет локальный манифест `backup_manifest.json`. Файлы удаляются из облака **только** если они уже локально скачаны, проверены по хэшу (SHA-256) и их возраст превышает `DAYS_TO_KEEP_IN_CLOUD`.
- **Идемпотентность**: Безопасно продолжает работу после прерывания сессии. Пропускает уже сохраненные файлы, дедуплицирует загрузки и корректно обрабатывает одиночные фото, входящие в уже скачанные дневные группы.
- **Обход защиты Google**: Использует стандартный движок браузера, скрывает флаги автоматизации и умеет обходить ограничения React Virtual DOM.

## Установка

Требуется установленный Python 3.

```powershell
# Создание и активация виртуального окружения
python -m venv venv
venv\Scripts\activate

# Установка зависимостей
pip install -r requirements.txt
playwright install chromium
```

Если используется системный Chrome, скрипт запускает канал `chrome`.

## Настройка (config.json)

Создайте локальный `config.json` на основе `config.example.json`.

Ключевые параметры:

- `DEST_DIR`: корневая папка для архива.
- `PROFILE_DIR`: профиль браузера с Google-сессией.
- `TEMP_DIR`: временная папка для загрузок и распаковки.
- `STATE_FILE`: manifest состояния, по умолчанию `backup_manifest.json`.
- `DAYS_TO_KEEP_IN_CLOUD`: сколько дней фото остаются в Google Photos.
- `ALLOW_CLOUD_DELETE`: включает реальное удаление из облака.
- `DRY_RUN_DELETE`: оставляет удаление выключенным, но показывает, что было бы удалено.
- `MAX_DELETE_BLOCKS_PER_RUN`: лимит удаляемых блоков за запуск, `0` означает без лимита.
- `LOG_LEVEL`: уровень подробности логов. `WARNING` пишет старт, итоговый отчет и проблемы; `INFO` добавляет скачивания, удаления, dry-run и важные решения по облаку; `DEBUG` включает техническую механику UI, скролла, чекбоксов и найденных блоков.
- `SCROLL_STEP_PIXELS`: шаг прокрутки Google Photos. Значение `500` оставляет заметный нахлест между экранами и снижает риск пропуска редких одиночных элементов.
- `SCROLL_WAIT_MS`: пауза после прокрутки, чтобы Google Photos успел подгрузить новые элементы.

## Запуск

```powershell
venv\Scripts\activate
python google_photos_bot.py
```

Или через `start_bot.bat`.

При первом запуске может потребоваться ручной вход в Google. Сессия сохраняется в `bot_profile`.

## Отчет по manifest

`manifest_report.py` читает только локальные `config.json` и `backup_manifest.json`. Он не запускает браузер, ничего не скачивает и не удаляет.

```powershell
python manifest_report.py
python manifest_report.py --date 2025-11-11
python manifest_report.py --unverified
python manifest_report.py --errors
python manifest_report.py --delete-candidates
python manifest_report.py --as-of 2026-05-17 --delete-candidates
```

Полезные режимы:

- `--date YYYY-MM-DD`: показать все блоки конкретной даты.
- `--delete-candidates`: показать подтвержденные блоки старше `DAYS_TO_KEEP_IN_CLOUD`, без дублей одиночных item-записей, покрытых группами.
- `--all-delete-candidates`: показать сырые candidate-записи, включая item-записи, покрытые группами.
- `--blocked-delete`: показать старые блоки, которые нельзя удалять, потому что архив не подтвержден.
- `--limit N`: увеличить количество строк в списках.

## Важная логика безопасности и предотвращения потери данных

`DAYS_TO_KEEP_IN_CLOUD` определяет, какие блоки пора удалить из облака. Но само удаление разрешается только после локальной проверки файлов и записи `files_verified` в manifest.

Если блок старше порога, но локальная копия не подтверждена, скрипт сначала скачает и проверит блок. Если скачать или проверить не удалось, удаление не выполняется.

Если дневная группа успешно скачана архивом и проверена, одиночные фото этой даты считаются покрытыми этой группой и не скачиваются повторно.

После реального удаления скрипт ждет закрытия модального окна Google Photos и исчезновения исходного checkbox из DOM. Только после этого manifest получает `deleted_from_cloud=true`. Если удаляется групповая запись, дочерние item-записи, покрытые этой группой, тоже каскадно отмечаются удаленными в manifest.

Для первого запуска после перехода на manifest разумно временно поставить:

```json
"DRY_RUN_DELETE": true
```

Так можно собрать состояние и посмотреть отчет без риска очистки облака.
